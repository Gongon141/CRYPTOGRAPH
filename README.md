# cryptum — Multi-Algo Cryptotool

Консольная программа для шифрования и расшифрования данных с модульной
архитектурой: главный модуль отделён от алгоритмов, каждый алгоритм — отдельная
динамически подключаемая библиотека. Реализовано шесть шифров: **XOR**,
**Атбаш**, **Цезарь**, **Виженер**, **Плейфер**, **нигилист**.

Все файлы проекта лежат в одной папке (плоская структура).

## 1. Сборка

Нужен компилятор GCC (g++ 9.1 и новее). В папке проекта выполнить:

```
make
```

После сборки появится каталог `build/`: исполняемый файл `build/cryptum`,
тестовая утилита `build/cryptum_tests` и каталог `build/lib/` с библиотеками.

Альтернатива — сборка через CMake:

```
mkdir build && cd build && cmake .. && cmake --build .
```

## 2. Запуск без установки (самый простой способ)

Проще всего запускать программу прямо из папки проекта — она сама находит свои
библиотеки рядом, ничего настраивать не нужно:

```
./build/cryptum --help
./build/cryptum -a caesar -m generate-key -s key.bin
./build/cryptum -a caesar -m encrypt -k key.bin -i secret.txt -o secret.enc
./build/cryptum -a caesar -m decrypt -k key.bin -i secret.enc -o secret.txt
```

Этот способ работает всегда: без установки, без sudo и без настройки PATH.

## 3. Использование

Флаги командной строки (есть короткий и длинный вид):

- `-a`, `--algorithm` — алгоритм: xor, atbash, caesar, vigenere, playfair, nihilist
- `-m`, `--mode` — режим: encrypt, decrypt, generate-key
- `-k`, `--key` — файл ключа (или `-` для stdin)
- `-g`, `--generate-key` — сгенерировать ключ на лету при шифровании (сохраняется через `-s`)
- `-s`, `--save-key` — куда сохранить сгенерированный ключ (или `-` для stdout)
- `-i`, `--input` — входной файл (или `-` для stdin)
- `-o`, `--output` — выходной файл (или `-` для stdout)
- `-h`, `--help` — справка

Знак `-` вместо имени файла означает стандартный поток (stdin/stdout), что
позволяет объединять вызовы в конвейеры. Шифр Атбаша ключа не требует. Шифр
Цезаря использует ключ длиной 1 байт, Виженер — не менее 16 байт; остальные
принимают ключ любой длины. Нужную длину ключа программа учитывает сама в
режиме генерации ключа.

## 4. Тестирование

```
make test
```

Для каждого из шести шифров выполняется пять тестов по схеме
«зашифровать → расшифровать → сравнить с исходником».

## Замечание по реализации

Чтобы программа отвечала ТЗ (шифровать файлы любого типа без потерь и быть
полностью обратимой), все шифры работают с байтами:

- **XOR** — байт ⊕ байт ключа (ключ повторяется по кругу);
- **Атбаш** — зеркальное отражение значения байта: c = 255 − b;
- **Цезарь** — сдвиг байта на значение ключа (1 байт): c = (b + k) mod 256;
- **Виженер** — сдвиг байта на байт ключа (ключ от 16 байт): c = (b + k_i) mod 256;
- **Плейфер** — квадрат 16×16 (256 ячеек), три правила по парам байт; одинаковые
  пары и непарный последний байт не меняются (сохраняет длину и обратимость);
- **Нигилист** — квадрат 16×16, сложение координат байта данных и байта ключа
  по модулю 16.

Все преобразования не меняют длину данных и точно обратимы.

## Соответствие ТЗ и примечания

Выполнено в коде:

- модульная архитектура: главный модуль отделён от алгоритмов, каждый шифр — отдельная библиотека (п. 4.4.4.1);
- взаимодействие через C ABI со структурами `ConstBuffer` / `MutBuffer` и функциями `get_algorithm_info`, `get_output_size`, `encrypt`, `decrypt` (п. 4.4.7.1);
- данные внутри программы — только массивы байт, файлы и потоки в бинарном режиме (п. 4.1.2);
- три режима работы и валидация аргументов; приоритет у справки (п. 4.1.3, 4.1.4);
- GNU-аргументы с короткими и длинными псевдонимами (п. 4.1.5.2);
- генерация ключа криптостойким ГПСЧ нужной для алгоритма длины (п. 4.1.3.2);
- генерация ключа «на лету» при шифровании флагом `-g` (п. 4.1.3.3);
- выделение буферов на стороне главного модуля (п. 4.4.7.2);
- затирание буферов с ключом и открытым текстом перед освобождением (п. 4.4.7.3);
- библиотеки не бросают исключения наружу, сообщают об ошибках кодом (п. 4.4.7.4);
- главный модуль не делает предположений об алгоритме, берёт метаданные через `get_algorithm_info` (п. 4.4.7.5);
- платформозависимое имя библиотеки: `libxor.so` в Linux, `xor.dll` в Windows (п. 4.4.7.7);
- обработка ошибок через исключения C++ без аварийного завершения (п. 4.2);
- строгие флаги сборки `-Wall -Wextra -Werror`, C++17, CMake и Make (п. 4.4.8);
- отдельная тест-утилита, по 5 тестов на алгоритм «зашифровать → расшифровать → сравнить» (п. 4.4.10);
- скрипты установки и удаления (п. 4.4.6).

Примечания (вне рамок реализованных классических шифров или специфично для Windows):

- **Потоковая обработка больших файлов** (п. 4.3.2.1). Сейчас файл целиком читается в память. Для гарантии ≤128 МБ ОЗУ на файлах до 1 ГБ потребовалась бы поблочная обработка; для учебных данных РГР текущего подхода достаточно.
- **PKCS#7** (п. 4.1.2.4) не применяется: все шесть шифров — потоковые/подстановочные и не меняют длину данных, блочных алгоритмов среди них нет.
- **Вектор инициализации** (п. 4.4.7.6) не используется: ни один из реализованных шифров его не требует.
- **Windows** (п. 4.1.6.2, 4.4.6.2): сборка ориентирована на Linux (ТЗ это допускает). Для Windows архитектура уже готова — нужны загрузка через `LoadLibrary`, установка кодовой страницы консоли в UTF-8 и инсталлятор на Batch/PowerShell.
